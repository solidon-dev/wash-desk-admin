# Git 커밋 컨벤션

## 핵심 원칙

- 모든 커밋 메시지 제목은 한국어로 작성한다
- 타입(type)은 영문 소문자를 사용한다
- 하나의 커밋은 하나의 논리적 변경만 담는다
- **스코프(scope)는 기본적으로 사용하지 않는다** — 사용자가 명시적으로 "스코프 써라" 또는 "scope 사용해라"고 지시할 때만 사용한다

---

## 커밋 단위 전략

하나의 커밋은 하나의 논리적 변경만 담는다. 아래 기준을 따른다.

### 기본 원칙

- **기능 단위로 커밋한다** — 하나의 기능, 하나의 버그 수정, 하나의 리팩토링을 각각 별도 커밋으로 분리한다
- **파일 단위가 아니다** — 여러 파일이 수정되더라도 하나의 목적이면 하나의 커밋이다
- **목적이 다르면 반드시 분리한다** — 기능 추가와 버그 수정을 한 커밋에 섞지 않는다
- **동작하는 상태로 커밋한다** — 빌드가 깨지거나 미완성인 상태로 커밋하지 않는다

### 커밋 분리 기준

| 상황                                         | 처리 방법       |
| -------------------------------------------- | --------------- |
| 새 기능 추가 + 기존 버그 수정                | 커밋 2개로 분리 |
| 여러 파일에 걸친 하나의 기능 구현            | 커밋 1개        |
| 리팩토링 + 기능 변경                         | 커밋 2개로 분리 |
| 설정 파일 변경 + 코드 변경                   | 커밋 2개로 분리 |
| 의존성 추가 + 그 의존성을 사용하는 기능 구현 | 커밋 2개로 분리 |
| 동일한 목적의 스타일·포맷 정리               | 커밋 1개        |

### 커밋 크기 기준

- **너무 작게**: `파일 1개, 한 줄 수정` 수준의 커밋은 의미 있는 단위로 묶는다
- **너무 크게**: 하루치 작업을 한 커밋에 몰아넣지 않는다
- **적당한 크기**: 커밋 메시지 제목 한 줄로 변경 내용을 명확히 설명할 수 있는 단위

---

## 커밋 메시지 형식

### 기본 형식 (스코프 없음 — 기본값)

```
<type>: <한국어 제목>

[본문 - 선택사항]
변경 이유나 주요 내용을 한국어로 간략히 작성

[푸터 - 선택사항]
BREAKING CHANGE: 설명
또는 이슈 참조: #123
```

### 스코프 사용 형식 (사용자 지시 시에만)

```
<type>(<scope>): <한국어 제목>
```

---

## 타입 (Type)

| 타입     | 설명                            |
| -------- | ------------------------------- |
| feat     | 새로운 기능 추가                |
| fix      | 버그 또는 오작동 수정           |
| style    | 동작 변경 없는 포맷·스타일 변경 |
| refactor | 동작 변경 없는 코드 구조 개선   |
| perf     | 성능 개선                       |
| test     | 테스트 추가 또는 수정           |
| chore    | 빌드·설정·의존성·환경 관련 작업 |
| docs     | 문서 작성 또는 수정             |
| revert   | 이전 커밋 되돌리기              |
| ci       | CI/CD 파이프라인 설정 변경      |

## 스코프 (Scope) — 사용자 지시 시에만 활성화

스코프는 변경이 영향을 미치는 영역을 나타낸다.
프로젝트 성격에 맞게 자유롭게 정의하면 되며, 아래는 분야별 참고 예시다.
스코프 명칭이 아래 목록에 없어도 프로젝트 도메인에 맞게 자유롭게 만들어 쓴다.

| 스코프   | 설명                                    | 적용 분야         |
| -------- | --------------------------------------- | ----------------- |
| init     | 초기 프로젝트 세팅                      | 모든 프로젝트     |
| core     | 핵심 로직 / 엔진                        | 모든 프로젝트     |
| config   | 설정 파일                               | 모든 프로젝트     |
| build    | 빌드 시스템 (Makefile, CMake 등)        | 모든 프로젝트     |
| types    | 타입 / 자료구조 정의                    | 모든 프로젝트     |
| utils    | 유틸리티 함수                           | 모든 프로젝트     |
| test     | 테스트 코드                             | 모든 프로젝트     |
| deps     | 외부 의존성 추가·변경·제거              | 모든 프로젝트     |
| auth     | 인증 / 보안 / 권한                      | 소프트웨어        |
| api      | API 인터페이스 및 통신                  | 소프트웨어        |
| db       | 데이터베이스 / 저장소                   | 소프트웨어        |
| ui       | 사용자 인터페이스                       | 소프트웨어        |
| cli      | 커맨드라인 인터페이스                   | 소프트웨어        |
| driver   | 하드웨어 드라이버                       | 임베디드 / 시스템 |
| hal      | 하드웨어 추상화 계층 (HAL)              | 임베디드          |
| firmware | 펌웨어 로직                             | 임베디드          |
| bsp      | 보드 지원 패키지 (BSP)                  | 임베디드          |
| isr      | 인터럽트 서비스 루틴                    | 임베디드          |
| rtos     | RTOS 태스크 / 스케줄러                  | 임베디드          |
| protocol | 통신 프로토콜 (UART, SPI, I2C, MQTT 등) | 임베디드 / IoT    |
| sensor   | 센서 입력 처리                          | IoT               |
| network  | 네트워크 / 무선 통신                    | IoT / 소프트웨어  |
| power    | 전원 관리 / 저전력 처리                 | 임베디드 / IoT    |
| mem      | 메모리 관리                             | 시스템 / 임베디드 |
| sync     | 동기화 / 뮤텍스 / 세마포어              | 시스템 / 임베디드 |

---

## 제목 작성 규칙

1. 한국어로 작성한다
2. 명령형 어미를 사용한다 — 추가, 수정, 제거, 개선, 변경, 초기화, 분리 등
3. 마침표를 붙이지 않는다
4. 50자 이내로 간결하게 작성한다
5. 무엇을 했는지 명확하게 표현한다

## 분야별 예시

### 소프트웨어 / 웹 / 앱

```
feat: JWT 기반 토큰 갱신 로직 추가
fix: 사용자 목록 조회 시 빈 배열 반환 오류 수정
style: 버튼 컴포넌트 간격 및 폰트 크기 조정
refactor: 전역 상태 관리 로직 분리
chore: axios 1.6.0으로 업그레이드
```

### 임베디드 / 펌웨어

```
feat: SPI 플래시 메모리 드라이버 초기 구현
fix: UART 수신 인터럽트 핸들러 버퍼 오버플로 수정
perf: DMA 전송 시 불필요한 메모리 복사 제거
feat: 센서 수집 태스크 우선순위 조정
chore: STM32F4 보드 핀맵 초기 설정 추가
```

### IoT / 통신

```
feat: MQTT 브로커 연결 및 재연결 로직 구현
fix: DHT22 온습도 센서 읽기 타이밍 오류 수정
feat: Wi-Fi 연결 실패 시 자동 재시도 추가
perf: 슬립 모드 진입 조건 최적화
feat: Modbus RTU 슬레이브 응답 파싱 추가
```

### 시스템 / 드라이버

```
feat: PCIe NVMe 컨트롤러 드라이버 기본 구조 추가
fix: 커널 힙 해제 시 이중 해제 버그 수정
perf: 스핀락 경합 구간 축소로 컨텍스트 스위칭 감소
refactor: 플랫폼 종속 코드 HAL 레이어로 분리
docs: 드라이버 API 레퍼런스 주석 추가
```

### 공통 / 초기화

```
chore: 프로젝트 초기 구조 및 빌드 환경 설정
chore: .gitignore 및 에디터 설정 파일 추가
docs: 프로젝트 README 초안 작성
chore: GitHub Actions 빌드 파이프라인 추가
```

## 본문 작성 규칙 (선택사항)

- 제목만으로 설명이 부족할 때 작성한다
- 왜 변경했는지를 중심으로 작성한다
- 한 줄에 72자 이내로 작성한다
- 제목과 본문 사이에 빈 줄을 넣는다

## AI 커밋 작성 원칙

AI가 커밋을 생성할 때는 아래 원칙을 반드시 따른다.

1. 커밋 메시지 제목은 항상 한국어로 작성한다
2. 타입은 위 타입 목록에서만 선택한다
3. **스코프는 사용자가 명시적으로 지시하기 전까지 절대 사용하지 않는다**
4. 사용자가 "스코프 써라", "scope 붙여라" 등 명시적으로 지시한 경우에만 스코프를 추가하며, 스코프 명칭은 위 목록을 참고하되 프로젝트 도메인에 맞게 자유롭게 정의한다
5. 변경된 파일과 내용을 기반으로 정확하고 구체적인 제목을 작성한다
6. 여러 기능을 하나의 커밋에 묶지 않는다 — 논리적 단위로 커밋을 분리한다
7. "기타 수정", "여러 파일 수정", "작업 중" 같은 모호한 표현을 절대 쓰지 않는다
8. 사용자가 커밋 단위나 메시지를 직접 지정하면 그것을 최우선으로 따른다
